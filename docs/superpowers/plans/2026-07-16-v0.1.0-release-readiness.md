# GoAgents v0.1.0 多模块发布就绪实施计划

> 基于 `docs/superpowers/specs/2026-07-16-v0.1.0-release-readiness-design.md` 执行。

**目标：** 在不打 tag、不推送的前提下，完成单仓模块路径迁移和发布元数据收敛，并留下
可重复的静态发布布局门禁。

## Task 1：建立先失败的发布布局门禁

新增 `scripts/verify-release-layout.sh`，显式列出 12 个发布模块和 5 个示例模块，检查：

- module path 与单仓路径一致；
- 发布模块内部依赖为 `v0.1.0`；
- 发布模块不含内部 `replace`；
- 示例模块路径一致且本地 replace 仍指向仓库内模块；
- 不存在旧的 `github.com/eruca/<module>` module/require/replace 路径。

先在当前 checkout 运行，预期因旧 module path 失败，证明门禁能够捕获现有阻塞。

## Task 2：迁移模块与导入路径

对以下已知模块前缀做一次机械、边界明确的迁移：

```text
github.com/eruca/<module>
-> github.com/eruca/goagents/<module>
```

范围包括 `go.mod`、Go import、README、契约和仍作为当前说明使用的设计文档；不得改变第三方
依赖、业务逻辑或 API 名称。迁移后运行 `gofmt` 和旧路径扫描。

## Task 3：收敛内部版本与 replace

- 所有内部 `require` 改为 `v0.1.0`；
- 从 6 个有内部依赖的发布模块移除相对 `replace`；
- 5 个示例模块保留并更新本地 `replace`；
- 根 `go.work` 为尚未发布的内部 `v0.1.0` 增加版本级本地映射；
- 独立模块直接运行 `go mod tidy`，发布消费者用临时 `-modfile` 注入本地替换后运行
  `GOWORK=off go mod tidy -diff`，核对没有无关依赖漂移。

## Task 4：更新发布文档

更新 `docs/modules.md`、各模块发布说明和根 README：

- 区分 12 个发布模块与 5 个验证模块；
- 记录准确 module path、tag 路径和发布顺序；
- 明确 remote、许可证和 tag 后 clean-consumer 验证仍未完成。

## Task 5：验证与审查

依次运行：

```bash
bash scripts/verify-release-layout.sh
bash scripts/verify-all.sh
git diff --check
```

再运行 Host 黑盒门禁，确认模块路径迁移没有破坏实际组合。最后做独立代码审查；只有
Critical=0、Important=0 才按语义提交并合并 `main`。

真实 Provider 已在同一产品代码基线上通过，若本阶段不改 Host 业务代码，可不重复调用外部模型；
发布前最终 tag commit 仍需重新跑完整验收。
