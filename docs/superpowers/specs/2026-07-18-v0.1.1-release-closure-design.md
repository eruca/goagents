# GoAgents Host Lifecycle v0.1.1 发布收口设计

**日期：** 2026-07-18

**状态：** 已批准，待实施

## 1. 决策摘要

本阶段把已经进入 `main` 的 Host 生命周期硬化收敛为一次可审计、不可变的产品发布。
产品里程碑称为 **GoAgents Host Lifecycle v0.1.1**，但继续遵守单仓多 Go module 的独立
版本语义，只发布真正发生公开变化的模块：

| 模块 | 当前公开 tag | 本次 tag | 原因 |
|---|---|---|---|
| `hostkit` | 无 | `hostkit/v0.1.0` | 新增标准库 Host 生命周期契约 |
| `workflowkit` | `workflowkit/v0.1.0` | `workflowkit/v0.1.1` | SQLite `Update` 改为事务内读改写，避免并发终态丢失 |
| `runkit` | `runkit/v0.1.0` | `runkit/v0.1.1` | 新增精确失败 pending approval checkpoint 的可选能力 |

其余十个已发布模块没有公开代码变化，继续保持各自的 `v0.1.0`，不创建无内容的
`v0.1.1` tag。

GitHub 只创建一个产品 Release：

- 主 tag：`hostkit/v0.1.0`；
- 标题：`GoAgents Host Lifecycle v0.1.1`；
- 正文同时列出 `workflowkit/v0.1.1` 和 `runkit/v0.1.1`。

发布采用两阶段不可变流程：先把三个 annotated tags 固定到同一个 release candidate
commit，再于发布完成后用新提交回写远端证据。已经公开的 tag 永不移动、覆盖或删除。

## 2. 当前基线

设计时的已核验基线为：

- canonical remote：`git@github.com:eruca/goagents.git`；
- `main` 与 `origin/main`：`31bb6b899b21bbd48b8534fa325cf058a23102e6`；
- 该 commit 的 GitHub CI 已成功；
- v0.1.0 的十二个 annotated module tags 都 peel 到
  `8bb91d8889318f98993644c380011cb707d19e45`；
- 根仓仍刻意不创建裸 `v0.1.0` tag；
- 当前仓库共有十三个发布模块和五个只验证、不发布的示例模块；
- `hostkit` 已进入模块布局和 workspace，但尚无公开 tag；
- v0.1.0 后发生公开模块代码变化的范围只有 `hostkit`、`workflowkit` 和 `runkit`；
- Host 启动错误分类、`SIGINT/SIGTERM` drain、强制收口、重启恢复和外部副作用边界均已有
  真实进程测试；
- GitHub Actions 已升级到 Node.js 24 runtime 对应版本，Apache-2.0 门禁已固定
  SHA-256；
- 根 README 和生命周期设计中的部分状态文字仍停留在“两个 P2 未关闭”或“待实施”，需要
  在 release candidate 中同步当前事实。

当前远端状态、CI、tag 和 Release 都属于易变外部事实。实施时必须重新读取，不能把本节
快照直接当作发布证据。

## 3. 目标

- 用逐模块版本清单替代 release gate 中的全局单一版本假设；
- 让 workspace 和 Host 示例准确声明本次三个模块版本；
- 同步当前入口文档，但保留 v0.1.0 验收记录的历史事实；
- 对同一个 release candidate commit 完成完整、无 `SKIP` 的发布门禁；
- 使用 atomic push 一次推送三个 annotated tags；
- 用正式 tag 的干净外部消费者证明 Go module proxy 路径可用；
- 创建单一 GitHub Release 并明确产品里程碑、模块版本和运行边界；
- 在发布后把 exact commit、tags、CI、Release 和 consumer 证据写回仓库；
- 保证发布记录不包含任何敏感值或本机临时信息。

## 4. 非目标

- 不修改 `hostkit`、`workflowkit`、`runkit` 或 Host API 的运行时行为；
- 不为验收测试添加硬编码绕过、特殊分支或降低断言；
- 不实现 Skill 动态激活、脚本执行、远程 registry 或依赖安装；
- 不实现多 Worker、跨进程 lease、分布式执行或 crash supervision；
- 不扩大 graceful shutdown 为任意崩溃点 exactly-once 保证；
- 不为没有公开代码变化的十个模块创建新 tag；
- 不创建根 `v0.1.1` tag；
- 不改写 v0.1.0 发布时的历史判断和证据。

若验收发现运行时缺陷，发布立即停止；缺陷修复必须另行做最小设计、测试和语义提交，
不能在本发布收口切片内静默修改行为。

## 5. Release Candidate

### 5.1 逐模块版本清单

`scripts/verify-release-layout.sh` 不再使用一个全局 `VERSION` 推断所有模块版本。发布清单为
每个模块明确记录目录、module path、当前有效版本和 tag：

| 发布模块 | 有效版本 |
|---|---|
| `goagent` | `v0.1.0` |
| `hostkit` | `v0.1.0` |
| `artifactkit` | `v0.1.0` |
| `contextkit` | `v0.1.0` |
| `evalkit` | `v0.1.0` |
| `ocrs` | `v0.1.0` |
| `workflowkit` | `v0.1.1` |
| `llmkit` | `v0.1.0` |
| `mcpkit` | `v0.1.0` |
| `runkit` | `v0.1.1` |
| `skillkit` | `v0.1.0` |
| `workflowkit/agentstep` | `v0.1.0` |
| `mcpkit/officialsdk` | `v0.1.0` |

门禁不能再把所有内部 `require` 强制为同一版本，也不能用被依赖模块的最新版本覆盖
未重新发布模块的历史依赖。发布清单需要另外记录显式的逐依赖边版本矩阵。例如：

- `runkit` 继续依赖未变化的 `goagent v0.1.0`；
- `workflowkit/agentstep` 继续依赖其发布时声明的 `workflowkit v0.1.0`；
- `mcpkit/officialsdk` 继续依赖 `goagent v0.1.0` 和 `mcpkit v0.1.0`；
- 新增的 `hostkit` 不依赖任何 GoAgents 模块。

逐依赖边矩阵必须与各模块本次实际发布内容一致；只有重新发布某个依赖方模块时，才允许
调整它的内部 `require`。发布增量另设固定集合，只允许本次三个目标 tag。布局门禁只验证
候选元数据，不把尚未创建的远端 tag 伪装为已存在；远端 tag 检查属于发布阶段的外部门禁。

### 5.2 Workspace 与 Host 示例

根 `go.work` 的精确 replacement 使用逐模块有效版本：

- `github.com/eruca/goagents/hostkit v0.1.0 => ./hostkit`；
- `github.com/eruca/goagents/workflowkit v0.1.1 => ./workflowkit`；
- `github.com/eruca/goagents/runkit v0.1.1 => ./runkit`；
- 其他模块继续使用 `v0.1.0`。

`examples/host-api/go.mod` 同步要求：

- `hostkit v0.1.0`；
- `workflowkit v0.1.1`；
- `runkit v0.1.1`；
- 其他内部模块继续使用 `v0.1.0`；
- 现有仓内相对 `replace` 保留，因为该 module 只用于组合与验收，不创建 tag。

其他示例模块若不使用本次新增公开能力，不做无意义版本迁移。发布模块中也不新增本地
`replace`。

### 5.3 文档状态

release candidate 只同步当前状态，不改写历史：

- 根 README 不再把两个已关闭 P2 表述为“当前已接受问题”，并链接新的 v0.1.1
  发布就绪记录；
- `2026-07-17-host-lifecycle-hardening-design.md` 状态改为“已实现，待发布”；
- 新增 `2026-07-18-v0.1.1-release-readiness.md`，记录版本矩阵、验收范围和当前
  candidate 状态；
- v0.1.0 最终验收、发布就绪和发布记录继续保留当时的两个 P2，不反向修改历史结论。

候选文档不能包含其自身尚未生成的 commit SHA。candidate commit 创建后，不再修改其内容；
后续所有门禁在外部绑定该 exact SHA，最终由发布后证据记录写入精确值，避免自引用 commit。

### 5.4 候选边界

candidate 只允许以下变更：

- 发布版本和布局门禁；
- `go.work` 与 Host 示例的版本声明；
- 当前入口文档和新的发布就绪记录；
- 本设计和后续实施计划。

不得夹带运行时代码、测试行为修改、无关格式化或重构。候选形成后先做独立 diff review；
只有 `Critical=0`、`Important=0` 才能进入外部发布门禁。

## 6. 发布门禁

所有必需门禁必须在同一个干净 candidate checkout 上执行。任何新提交都会使之前绑定旧
SHA 的证据失效。

### 6.1 仓库与模块门禁

- `bash -n scripts/verify-release-layout.sh`；
- `bash scripts/verify-release-layout.sh`；
- `GOPROXY=off bash scripts/verify-all.sh`；
- 发布模块的默认测试、race 和 vet；
- `git diff --check`；
- LICENSE 标准标题和固定 SHA-256；
- CI workflow 固定使用已批准的 Actions major；
- 工作区没有未跟踪或未提交文件。

离线门禁只在依赖已准备完整的环境运行。缺失依赖必须显式报告为环境阻塞，不能临时改
`go.mod`、放宽门禁或以联网下载结果代替离线复现。

### 6.2 Host 真实进程门禁

必须覆盖：

- 正常 signal drain 和同 runtime home 重启；
- drain timeout 后活动 workflow 的失败收口；
- 第二次 signal 立即进入强制阶段；
- 配置失败的结构化错误和稳定退出码；
- 监听失败的结构化错误和稳定退出码；
- cleanup helper 不能用强杀掩盖被测进程未退出；
- shutdown 后 terminal AgentRun、approval checkpoint 和 workflow 状态一致；
- 外部副作用已提交与未提交两个边界；
- operator 显式 requeue，且系统不自动重放；
- 既有 MVP 黑盒闭环不回归。

这些测试必须继续使用真实 Host 子进程、SQLite、loopback Provider/OIDC 和隔离
`.smoke.` Keychain 标识。不得访问或删除生产默认 Keychain 项。

### 6.3 真实 Provider 门禁

必须执行：

- `TestRealProviderMVPAcceptance`；
- `TestHostAPIProcessRealProviderLocalTrial`；
- `TestRealProviderTrialSensitiveValuesIncludesModel`；
- 与本次 shutdown 路径相关的输出泄漏扫描。

必需配置缺失、Keychain 不可用或测试主动 `SKIP` 都是阻塞，不是成功。日志和记录不得输出
endpoint、模型标识、API key、token、Prompt、模型响应、随机 observation 或原始 Provider
payload。

### 6.4 Candidate 干净消费者

candidate 推送到 canonical remote 且工作树干净后，在三个彼此隔离的全新临时 module 中：

1. 设置 `GOWORK=off`；
2. 禁止本地 `replace`；
3. 分别按 candidate exact commit 获取 `hostkit`、`workflowkit` 和 `runkit`；
4. 编译一个最小、真实使用公开 API 的 consumer；
5. 运行 `go mod tidy`、`go test ./...` 和 `go list -m all`；
6. 核对解析结果来自 canonical remote 的 candidate pseudo-version，而不是 workspace 或
   module cache 中的本地替换。

三个模块必须独立验证，不能用一个 consumer 的成功推断另外两个模块可发布。

### 6.5 远端 CI

candidate 必须先推送 `main`，再等待 GitHub CI 对该 exact SHA 返回 `success`。旧 commit 的
成功 CI 不可复用。CI 为 `queued`、`in_progress`、`cancelled`、`skipped`、`neutral`、
`timed_out` 或 `failure` 时都不得创建 tag。

## 7. Tag 发布

### 7.1 发布前检查

创建 tag 前重新核对：

- 本地 `HEAD`、`origin/main`、candidate SHA 三者完全相等；
- 工作区干净；
- 所有本地与远端门禁绑定同一 SHA；
- 本地和远端都不存在三个目标 tag；
- 不存在以 `hostkit/v0.1.0` 为 tag 的 GitHub Release；
- canonical remote URL 未变化。

### 7.2 原子发布

在本地创建三个 annotated tags：

```text
hostkit/v0.1.0
workflowkit/v0.1.1
runkit/v0.1.1
```

创建后逐个检查对象类型为 `tag`、注释非空、peeled commit 等于 candidate。随后使用一次
atomic push 把三个 tag 推到 canonical remote。若服务端不支持 atomic push，发布停止，
不得自动退化为三个顺序 push。

push 返回后必须重新读取远端 refs，确认三个 annotated tag 及其 peeled commit 都正确。
本地命令成功但远端读取不一致时，按远端事实处理，不继续创建 Release。

### 7.3 正式 tag 消费验证

远端 tag 核验后，重新创建三个全新临时 consumer：

- 分别获取 `hostkit@v0.1.0`、`workflowkit@v0.1.1`、`runkit@v0.1.1`；
- 保持 `GOWORK=off`、无本地 `replace`；
- 运行 `go mod tidy`、`go test ./...`、`go list -m all`；
- 核对 module path、版本和依赖图精确匹配版本清单。

Go proxy 尚未传播可以在固定总时限内有界重试；不得以 direct/local replace 绕过正式 tag
验证。总时限后仍不可用即停止发布。

## 8. GitHub Release

只有正式 tag consumer 全部通过后，才创建单一非 draft、非 prerelease 的 GitHub Release：

- tag：`hostkit/v0.1.0`；
- 标题：`GoAgents Host Lifecycle v0.1.1`；
- 列出三个 module/tag；
- 说明启动错误不再 panic、signal-aware drain、超时强制收口、重启恢复和外部副作用边界；
- 说明 GitHub Actions 与 Apache-2.0 SHA-256 两个维护项；
- 给出三个模块的安装命令；
- 链接 exact candidate CI；
- 明确仍是单 Host，不支持多 Worker 或分布式执行；
- 明确不承诺任意崩溃点 exactly-once；
- 明确不包含 Skill 动态激活。

Release 正文只使用稳定标识和公开 URL，不复制本地绝对路径或敏感验收输出。

## 9. 发布后证据回写

发布完成后新增
`docs/superpowers/specs/2026-07-18-v0.1.1-publication.md`，至少记录：

- candidate exact SHA；
- 三个 annotated tag 及 peeled SHA；
- candidate CI run URL、ID 和 `success`；
- candidate 与正式 tag consumer 的模块、版本和 PASS 结果；
- Host 真实进程、真实 Provider、无 `SKIP` 和敏感输出扫描结论；
- GitHub Release URL、主 tag、标题、发布时间；
- 保留的单 Host、外部副作用和非 exactly-once 边界；
- 发布期间是否发生有界重试及其非敏感原因。

根 README 从发布就绪记录切换到该发布记录。证据提交不移动既有 tags；推送后必须等待该
提交自己的 CI 成功。该 CI 证明发布记录没有破坏 `main`，不替代 candidate CI。

## 10. 失败与恢复

- 任一测试失败、必需测试 `SKIP`、独立审查未通过、CI 非成功、敏感输出、版本漂移或脏
  工作区都会阻止 tag 创建。
- candidate 修复后会产生新 SHA，旧 SHA 的测试、review、consumer 和 CI 证据全部失效。
- atomic tag push 报错后先读取远端三个 refs；只有确认三者都不存在时，才允许在原因明确
  后重试同一次 atomic push。
- 远端 tag 一旦存在，禁止 force-push、移动、覆盖或删除。
- tags 正确但 GitHub Release 创建失败时，从已核验 tags 继续重试 Release，不重建 tag。
- 正式 tag consumer 若暴露真实代码或元数据缺陷，停止创建 Release；已有 tag 保持不动，
  重新设计更高补丁版本，不能用本地 replace 或删除 tag 掩盖问题。
- 若 GitHub Release 已创建而证据回写失败，Release 和 tags 保持不动，只修复证据分支并
  重跑其 CI。
- 网络、Go proxy 或 GitHub API 的瞬时错误只允许有界重试；权限、版本冲突和内容不一致
  不是瞬时错误。

## 11. 完成定义

只有同时满足以下条件，才能宣布 GoAgents Host Lifecycle v0.1.1 发布完成：

1. release candidate 只包含批准范围内的元数据和文档收口；
2. 逐模块版本、workspace replacement 和 Host 示例依赖一致；
3. 完整 workspace、race、vet、release-layout、真实 Host、真实 Provider 和敏感输出门禁
   全部通过且无必需 `SKIP`；
4. candidate exact SHA 的独立审查为 `Critical=0`、`Important=0`；
5. candidate exact SHA 的 GitHub CI 为 `success`；
6. candidate pseudo-version 的三个干净 consumer 全部通过；
7. 三个 annotated tags 经 atomic push 且都 peel 到 candidate；
8. 三个正式 tag 的干净 consumer 全部通过；
9. `GoAgents Host Lifecycle v0.1.1` GitHub Release 已发布；
10. 发布后证据记录已推送，且其 CI 为 `success`；
11. 本地与远端没有未解释的分支、tag、Release 或敏感数据残留。
