## 1. 当前结论

Host 生命周期硬化和两个延期维护项已经进入 `main`，但本文不声称 tags 或 GitHub Release
已经创建。包含本文的最终 candidate commit 只有在完整本机门禁、独立复审、精确 SHA 的
远端 CI 和 candidate clean-consumer 全部通过后，才允许进入 tag 发布。

本文刻意不记录自身尚未生成的 commit SHA。发布完成后由独立 publication record 写入
candidate exact SHA 和全部外部证据，避免自引用 commit。

## 3. 必需门禁

- 逐模块 release-layout、完整 workspace、race、vet 和 `git diff --check`；
- Host 优雅 drain、超时、第二 signal、启动错误、重启和外部副作用真实进程门禁；
- 真实 Provider MVP 与真实 Provider × Host 本机门禁，无必需 `SKIP`；
- endpoint、model、credential、token，以及随机 Prompt、工具 observation、模型响应
  sentinel 的完整 stdout/stderr 零泄漏扫描；真实 Host artifact 必须先证明响应 sentinel
  到达最终模型结果。Host input artifact 只保存并证明 Prompt sentinel；临时、无工具 Skill
  prompt 经精确 `skill_refs` 的既有投影进入模型请求，并要求最终响应包含响应 sentinel；
- candidate exact 40 位 SHA 的三个物理路径位于当前 repo root 外的 `GOWORK=off`
  clean-consumer；解析出的
  pseudo-version 12 位 revision 后缀必须等于该 SHA 的前 12 位；
- candidate exact SHA 的 GitHub CI `success`；
- 独立审查 `Critical=0`、`Important=0`。

任一真实 Provider/Host harness 修复都会产生新的 candidate SHA；修复前的 Task 4、review、
consumer 和 CI 证据均失效，必须从 Task 4 Step 1–7 重跑。

## 4. 发布方式

三个 annotated tags 必须通过一次 atomic push 指向同一个 candidate。正式 tag consumer
通过后，以 `hostkit/v0.1.0` 创建标题为 `GoAgents Host Lifecycle v0.1.1` 的单一
GitHub Release。发布后用新的未打 tag 提交记录远端证据，既有 tags 永不移动。

## 5. 保留边界

本次仍是单 Host，不实现多 Worker、分布式执行或 Skill 动态激活。graceful shutdown
缩小受控 signal 的失败窗口，但不保证外部副作用与本地 checkpoint 在任意崩溃点
exactly-once；强制收口后的重试仍需 operator 核对外部副作用。
