以组合当前 checkout。

## 3. Tag 路径与顺序

不得创建根 `v0.1.0`。拟创建的子目录 tag 是：

```text
goagent/v0.1.0
artifactkit/v0.1.0
contextkit/v0.1.0
evalkit/v0.1.0
ocrs/v0.1.0
workflowkit/v0.1.0
llmkit/v0.1.0
mcpkit/v0.1.0
runkit/v0.1.0
skillkit/v0.1.0
workflowkit/agentstep/v0.1.0
mcpkit/officialsdk/v0.1.0
```

建议顺序：

1. 发布无内部依赖模块：`artifactkit`、`contextkit`、`evalkit`、`goagent`、`ocrs`、
   `workflowkit`，并先做全新 consumer 验证；
2. 发布依赖第一层：`llmkit`、`mcpkit`、`runkit`、`skillkit`、
   `workflowkit/agentstep`，再次做全新 consumer 验证；
3. 最后发布依赖 `mcpkit` 的 `mcpkit/officialsdk`，并验证其独立 consumer；
4. 任一层验证失败即停止，不继续发布后续 tag。

## 5. 仍阻塞公开发布的事项

1. 创建并配置与 module path 一致的 canonical remote：`github.com/eruca/goagents`；
2. 由项目所有者选择许可证并添加根 `LICENSE`；
3. 在包含许可证的最终 commit 上重跑布局、全仓和真实 Provider 门禁；
4. 先把最终 commit 推送到 canonical remote，再按依赖层发布 tag；每一层发布后，从空临时
   目录使用 `GOWORK=off`、无本地 replace 下载并编译该层代表模块；
5. 至少验证一个独立模块、一个第一层依赖模块和 `mcpkit/officialsdk`；只有每层
   clean-consumer 通过后才发布下一层，并在全部通过后创建 GitHub Release。

## 6. 已知限制

MVP 最终验收中的两个 P2 不变：Host 启动/监听错误仍使用 `panic`，CLI 仍无 signal-aware
graceful drain。本轮没有新增运行时缺陷；remote、许可证和 clean-consumer 属于发布门槛，
不应改写为产品 P0/P1。
